## 2. 目标

提供**基于 Supabase Auth** 的身份认证能力，覆盖**登录、密码管理、会话全生命周期管理与基础安全防护**：

- 客户端**只与本 HTTP 服务**交互，完成登录后获得 **access token** 与 **refresh token**（与 Supabase 会话语义一致），**不向客户端暴露** Supabase 项目 URL、密钥等实现细节。
- 登录方式采用**标识符（邮箱 / 手机号）与验证方式（密码 / OTP）正交组合**，再加上 **OAuth 第三方登录**，为用户提供灵活的登录选择。
- 支持 OTP 用户在已登录状态下**设置密码**，实现从无密码到有密码的平滑过渡。
- 提供**密码重置**作为密码登录的配套流程。
- 登录后提供**会话刷新、登出、当前用户查询**等完整会话管理能力。
- 对认证端点实施**基础频率限制**，防止暴力破解与滥用。
- 为第三方服务提供 **token 校验能力**（端点校验与自行 JWT 校验两种模式）。

在实现上**优先使用 Supabase Auth 已有数据库与服务**，**尽量少做额外业务开发**。

## 3. 核心范围

| 能力域 | 说明 |
| --- | --- |
| 登录与发令 | 支持**标识符 x 验证方式**正交组合 + OAuth 第三方登录（见 §4），登录成功后返回与 Supabase Auth 一致的会话 / 令牌结果（含 **access token** 与 **refresh token**）；对外响应形状由架构约定，见 `ARCHITECTURE.md`。 |
| 会话管理 | 提供 **token 刷新**、**登出**（撤销当前会话）、**获取当前用户信息** 三项能力，形成会话全生命周期闭环。 |
| 密码管理 | 提供**设置密码**（OTP 用户在已登录状态下创建密码）与**密码重置**（通过邮箱找回）两项能力，作为密码登录方式的配套。 |
| 统一入口 | 所有客户端与集成方**经本服务 HTTP API** 使用认证能力，由本服务封装 Supabase。 |
| Token 校验 | **支持**第三方调用**本服务**对 token 做身份校验（端点模式）；第三方也可**自行**使用 JWT secret 校验（自校验模式），**不强制**经本服务；两种模式并存，由第三方自选。 |
| 基础限流 | 对登录与 OTP 发送端点实施频率限制，防止暴力破解与短信/邮件滥发。 |

## 4. 登录方式

登录方式由**标识符**与**验证方式**正交组合，再加上 **OAuth 第三方登录**构成。

### 4.1 标识符 x 验证方式（凭证登录）

用户使用自有凭证（邮箱或手机号）配合密码或一次性验证码完成登录。两个维度正交组合，共 4 种方式：

| | 密码 | 一次性验证码（OTP） |
| --- | --- | --- |
| **邮箱** | 邮箱 + 密码 | 邮箱 + OTP |
| **手机号** | 手机号 + 密码 | 手机号 + OTP |

Supabase Auth 对以上 4 种组合均**原生支持**：

| 组合 | Supabase API | 说明 |
| --- | --- | --- |
| 邮箱 + 密码 | `signUp` / `signInWithPassword`（`email`） | 含注册与登录；已部分实现 |
| 手机号 + 密码 | `signUp` / `signInWithPassword`（`phone`） | 含注册与登录 |
| 邮箱 + OTP | `signInWithOtp`（`email`） | 无密码邮箱登录 |
| 手机号 + OTP | `signInWithOtp`（`phone`） | 需在 Supabase 侧配置短信 provider |

补充说明：

- 密码方式同时支持**注册**（首次使用即创建账号）和**登录**。
- OTP 方式同样支持首次使用时自动创建账号（Supabase Auth 默认行为）。
- 用户可以先通过 OTP 登录（无密码），后续在已登录状态下**设置密码**（见 §5），从而解锁密码登录方式。
- **密码注册开关**：邮箱+密码和手机号+密码的**注册**能力须通过**环境变量**控制是否开放。关闭时，密码注册端点返回拒绝响应；已有密码的用户**登录不受影响**。此开关不影响 OTP 登录、OAuth 登录以及已登录用户设置密码（§5）的能力。

### 4.2 OAuth 第三方登录

| # | 方式 | Supabase 支持情况 | 说明 |
| --- | --- | --- | --- |
| a | **Google** 集成登录 | 原生支持（`signInWithOAuth`） | 标准 OAuth 2.0 流程 |
| b | **微信**登录 | **无原生 provider** | 需自定义 OAuth / provider 集成 |

- **不要求**对 Google 账号强制绑定邮箱等业务规则（与 §7「用户与账号」一致）。
- 微信若 Supabase 官方无现成 provider，**允许**在需求范围内做**最小限度**的自定义 OAuth / provider 集成。

### 4.3 微信登录——多形式预留策略

微信登录在不同客户端场景下有三种形式，架构层面**预留全部三种形式的扩展空间**：

| 形式 | 场景 | 本期范围 |
| --- | --- | --- |
| 网页扫码登录 | 电脑端浏览器，页面展示二维码，用户手机微信扫码授权 | **本期实现** |
| APP 微信登录 | 移动端 APP 内拉起微信客户端授权 | 预留，本期不实现 |
| 微信内网页授权 | 微信浏览器内打开的网页，通过微信 JS-SDK 获取授权 | 预留，本期不实现 |

- **本期仅实现**电脑端网页扫码登录（微信开放平台「网站应用」）。
- 其余两种形式作为已知需求记录，后续按优先级迭代。
- 三种形式共享同一微信开放平台应用体系，实现时应考虑 provider 层面的可扩展性。

### 4.4 登录方式总览

本期需支持的全部登录方式共 **6 种**（4 种凭证组合 + 2 种 OAuth）：

| # | 方式 | 类型 |
| --- | --- | --- |
| 1 | 邮箱 + 密码 | 凭证登录 |
| 2 | 邮箱 + OTP | 凭证登录 |
| 3 | 手机号 + 密码 | 凭证登录 |
| 4 | 手机号 + OTP | 凭证登录 |
| 5 | Google OAuth | 第三方登录 |
| 6 | 微信网页扫码 | 第三方登录 |

## 5. 密码管理

| 能力 | 说明 | 本期范围 |
| --- | --- | --- |
| 设置密码 | 已登录用户（尚未设置密码）在已认证状态下创建密码，从而解锁密码登录方式 | **本期实现** |
| 密码重置 | 用户通过已注册邮箱发起重置请求，收到重置链接或验证码后设置新密码 | **本期实现** |
| 修改密码 | 用户已知当前密码，更换为新密码 | 不做，后续考虑 |

- **设置密码**：适用于通过 OTP 或 OAuth 首次登录、尚无密码的用户。调用时需携带有效的 access token（即已登录状态）。依赖 Supabase Auth 的 `updateUser({ password })` 内建能力。设置密码后，用户即可通过对应标识符（邮箱 / 手机号）+ 密码的方式登录。
- **密码重置**：依赖 Supabase Auth 的 `resetPasswordForEmail` + `updateUser` 内建能力。密码重置仅支持邮箱通道（与 Supabase Auth 行为一致）。
- **设置密码**与**密码重置**的区别：前者面向已登录但无密码的用户（主动设置）；后者面向忘记密码的用户（通过邮箱链接恢复）。

## 6. 会话管理

| 能力 | 说明 |
| --- | --- |
| 刷新 Token | 客户端使用 refresh token 换取新的 access token；与 Supabase `refreshSession` 语义一致。 |
| 登出 | 撤销当前会话；与 Supabase `signOut` 语义一致。仅撤销当前 session，**不做**全端登出。 |
| 获取当前用户 | 通过 access token 获取当前用户身份信息（用户 ID、邮箱等基础字段）；与 Supabase `getUser` 语义一致。 |

- 所有会话行为与 Supabase Auth 语义保持一致；对外响应形状由 `ARCHITECTURE.md` 约定。
- 会话持久化**完全依赖** Supabase Auth，本服务不额外存储 session 状态。

## 7. 用户与账号

- **不**要求用户必须拥有或绑定「已验证」的邮箱或手机号；**不**将「至少一个有效邮箱/手机」作为登录前提，**允许**在未满足该类条件时完成登录（具体用户模型以 Supabase Auth 为准）。
- 身份合并、修改邮箱/手机号、完整「账户中心」类能力**不在本期需求内**（见 §9）。

## 8. Token 校验（第三方集成）

为其他微服务或第三方系统提供用户身份校验能力，支持两种模式并存：

| 模式 | 说明 |
| --- | --- |
| **端点校验** | 本服务提供 HTTP 端点，第三方传入 access token，本服务返回校验结果（有效性 + 用户基础信息）。 |
| **自行校验** | 第三方使用 Supabase JWT secret 自行校验 token，不经本服务；本服务需在文档中说明 JWT 校验方式（公钥/secret、claims 结构、签名算法等）。 |

- 两种模式**不互斥**，第三方根据自身场景自行选择。
- 端点校验适用于不希望维护 JWT 校验逻辑的集成方；自行校验适用于对延迟敏感或希望去中心化的场景。

## 9. 安全与限流

- 对**登录端点**和 **OTP 发送端点**实施**基础频率限制**（如每 IP / 每账号的请求频率上限）。
- 限流策略以**防暴力破解**和**防短信/邮件滥发**为核心目标。
- 具体阈值由实现阶段根据 Supabase 自带限流能力决定，应用层做必要补充。
- **不在本期**实现完整的 WAF、IP 黑名单或高级风控系统。

## 10. 非目标（本期不做）

包括但不限于：

- **匿名登录**（Supabase 内建支持，但本期不纳入）。
- **多因素认证 MFA / 2FA**（Supabase 内建 TOTP 支持，但本期不纳入）。
- **认证事件通知**（Webhook / 消息队列通知其他微服务）。
- **微信 APP 登录**和**微信内网页授权**（架构预留，非本期交付）。
- **修改密码**（已知旧密码换新密码）。
- 显式**身份合并**、冲突处理等产品流程。
- 业务**用户中心**、资料管理。
- **修改邮箱**、修改手机号等账户维护流程。
- 复杂业务授权模型、本服务作为通用 **SSO** 提供方。
- **SSO / SAML** 企业集成。
- **对外 API 文档 / MCP 文档**作为交付物（可后续再加）。
- 在本需求文档中展开日志字段、错误码全集、正式安全/运维白皮书。

## 11. 技术约束（摘要）

- 实现栈与仓库约定见 `agents/ARCHITECTURE.md`（如 `bun`、`hono`、`zod`、`Supabase Auth` 等）。
- 认证状态与持久化**以 Supabase Auth / 托管库为主**；应用层**少造轮子**，非主链路第三方封装放在架构约定目录中。
- 微信登录的自定义 provider 实现放在 `src/providers/` 下，与主链路 Supabase 集成解耦。

## 12. 验收要点（服务级）

- 客户端**不直接依赖**暴露 Supabase 连接信息即可完成登录并取得令牌。
- **6 种登录方式**（4 种凭证组合 + 2 种 OAuth）均可在约定环境下完成登录并取得 access token + refresh token（微信仅验收电脑端网页扫码路径）。
- **设置密码**：通过 OTP 登录的用户可在已登录状态下成功设置密码，之后可使用密码方式登录。
- **密码重置**流程可正常走通（发起重置 → 收到邮件 → 设置新密码）。
- **会话管理**三项能力可正常工作：刷新 token、登出、获取当前用户信息。
- **Token 校验端点**可被第三方调用并返回正确的校验结果与用户信息。
- **限流**在登录与 OTP 发送端点生效，超出频率时返回适当的限流响应。
- 不包含第 10 节所列非目标范围内的强制验收。

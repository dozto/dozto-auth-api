> 它回答这个服务为什么存在、要做什么、不做什么，以及验收边界在哪里。

- 文档状态：`draft`

- 关联路线图：`agents/ROADMAP.md`

## 2. 为什么做

### 2.1 背景

需要一个独立的身份认证微服务，作为整个后台系统中的基础组件，统一承担用户认证、令牌签发与身份校验相关能力。这样可以把认证能力从各个业务服务中抽离出来，减少重复实现，并为客户端、第三方系统和 agent 提供统一的接入方式。

### 2.2 业务目标

- 建立一个可复用的统一身份认证中心，服务整个系统后台。
- 让其他业务服务不必重复实现用户认证与 token 处理逻辑。
- 为后续系统接入、第三方接入和 agent 接入提供稳定的认证基础能力。

### 2.3 用户目标

- 客户端可以通过标准化接口完成登录、会话建立、token 获取与刷新。
- 第三方服务可以方便地校验用户身份，而不必理解底层认证细节。
- 其他系统或 agent 可以通过清晰的 API / MCP 文档快速接入该服务。

## 3. 做什么

### 3.1 范围目标

- 提供通用的身份认证能力，而不是面向某一个具体业务场景定制。
- 优先复用 `betterauth` 的现成能力，尽量少改或不改其核心行为。
- 使用 `drizzle` 对接 `PostgreSQL`，形成稳定的数据持久化方案。

### 3.2 本期范围

- 提供面向客户端的认证 API。
- 提供 token 的签发、校验、刷新等基础能力。
- 支持邮件 + 验证码登录。
- 支持手机号码 + 验证码登录。
- 支持 Google 登录。
- 支持微信扫码登录。
- 支持 session 机制。
- 支持 refresh token 机制。
- 支持邮箱验证。
- 支持忘记密码。
- 支持修改邮箱。
- 支持修改手机号。
- 提供第三方服务自校验或通过 auth 服务校验用户身份的接入方式。
- 提供明确的 API 文档，供其他系统使用。
- 提供明确的 MCP 文档，供 agent 或自动化系统使用。
- 使用 `bun`、`hono`、`zod`、`drizzle`、`betterauth` 作为主技术栈完成实现。

### 3.3 关键角色

- 客户端：手机端、网页端等，直接调用认证服务。
- 第三方服务：消费认证服务签发的 token，或调用认证服务做身份校验。
- 平台内部其他服务：把认证服务作为后台微服务组件进行依赖。
- agent / 自动化系统：通过 API / MCP 文档理解和使用认证服务。

### 3.4 核心使用场景

- 用户在客户端完成登录并获得 token。
- 用户通过邮箱验证码完成登录。
- 用户通过手机号验证码完成登录。
- 用户通过 Google 完成第三方登录。
- 用户通过微信扫码完成网页登录。
- 用户首次通过邮箱验证码或手机号验证码登录时可以自动创建账号。
- 用户的本地主识别信息主要是邮箱或手机号。
- 用户首次通过 Google 或微信登录时，先创建独立身份，后续再通过显式流程合并到统一用户主体。
- 身份合并既支持用户主动发起，也支持后台或系统流程触发。
- 用户携带 token 访问第三方服务。
- 第三方服务自行校验 token 或调用认证服务完成校验。
- 内部系统或 agent 基于 API / MCP 文档调用认证能力。

### 3.5 功能需求

#### 3.5.1 认证核心能力

- FR-001：支持基于 `betterauth` 的用户认证流程。
- FR-002：支持 token 的生成、校验、刷新与失效处理。
- FR-003：尽可能直接使用 `betterauth` 官方能力，不做深度定制。
- FR-004：支持正式登录系统所需的完整账户机制，包括 session、refresh token、邮箱验证、忘记密码、修改邮箱、修改手机号。

#### 3.5.2 登录方式

- FR-005：支持邮件 + 验证码登录。
- FR-006：支持手机号码 + 验证码登录。
- FR-007：支持 Google 第三方登录。
- FR-008：支持微信扫码登录，当前一期优先支持网页扫码登录。
- FR-009：邮箱验证码首次登录允许自动创建账号。
- FR-010：手机号验证码首次登录允许自动创建账号。
- FR-011：邮箱和手机号是本地主要识别信息。
- FR-012：Google 和微信首次登录时先创建独立身份。
- FR-013：系统需要提供后续显式合并身份的能力。
- FR-014：身份合并既支持用户主动触发，也支持后台或系统流程触发。
- FR-015：身份冲突默认允许覆盖，但前提是必须先确认当前用户身份有效。

#### 3.5.3 客户端接入能力

- FR-016：提供面向客户端的标准 HTTP API。
- FR-017：返回格式、错误结构和调用方式应保持稳定且可文档化。

#### 3.5.4 第三方服务接入能力

- FR-018：支持第三方服务基于 token 完成用户身份认证。
- FR-019：支持第三方服务选择本地校验或委托 auth 服务校验。

#### 3.5.5 文档能力

- FR-020：提供清晰的 API 文档。
- FR-021：提供适合 agent 使用的 MCP 文档。

## 4. 不做什么

### 4.1 非目标

- 不把认证服务扩展成完整的业务用户中心。
- 不在第一阶段实现复杂的业务授权模型。
- 不为了少量特殊需求大幅改写 `betterauth` 内部机制。
- 不在当前阶段单独建设“auth 作为统一 SSO 提供方”的完整能力。
- 不承担其他业务服务自己的业务接口职责。

### 4.2 明确不做

- 不在当前文档中定义所有日志字段和错误码细节，这部分后续补充。
- 不在当前阶段展开非常正式的安全、运维、数据边界长文说明。
- 不在 service 或 controller 中直接写数据库访问逻辑。

## 5. 验收边界是什么

### 5.1 服务级验收

- 服务能够作为独立微服务组件存在于整个后台系统中。
- 客户端能够通过标准 API 完成基础认证流程并获得 token。
- 客户端能够通过邮件验证码完成登录。
- 客户端能够通过手机号码验证码完成登录。
- 客户端能够通过 Google 完成登录。
- 客户端能够通过微信扫码完成网页登录。
- 邮箱与手机号验证码首次登录时能够按预期自动创建账号。
- Google 与微信首次登录时能够创建独立身份并完成登录。
- 系统后续能够通过显式合并流程，把独立身份合并到统一用户主体。
- 身份合并既可以由用户主动发起，也可以由后台或系统流程触发。
- 身份冲突发生时，系统在确认当前身份有效后允许执行覆盖。
- session、refresh token、邮箱验证、忘记密码、修改邮箱、修改手机号等机制可正常工作。
- 第三方服务能够基于 token 进行用户身份校验。
- 第三方服务既可以选择自行校验，也可以通过 auth 服务完成校验。
- API 文档与 MCP 文档可供其他系统或 agent 使用。

### 5.2 Story 验收要求

- 每个 Story 必须有清晰、可测试的验收标准。
- Story 的验收标准应直接对应可观察的服务行为，而不是实现细节。
- 已完成的 Story 不直接改写新需求，变更通过新的 Epic / Story 承载。

### 5.3 Task 验收映射要求

- 每个 Task 必须归属于唯一一个 Story。
- Task 的测试或验证步骤必须能映射回 Story 的验收标准。
- Task 只描述执行与验证，不重新定义产品需求。
- 单元测试文件统一放在 `src/test/unit/`，文件名以 `.spec.ts` 结尾。
- 集成测试文件统一放在 `src/test/intg/`，文件名以 `.intg.ts` 结尾。
- `test:unit`、`test:intg` 与 `test` 作为标准测试命令入口，用于支撑 Task 验收与回归验证。

## 6. 有哪些业务、产品、合规约束

- 业务约束：该服务是整个后台系统中的基础认证微服务，需要对其他服务保持清晰边界。
- 产品约束：优先提供通用认证能力，不优先做强业务定制。
- 技术产品约束：认证能力优先复用 `betterauth`，尽量少改或不改。
- 数据约束：数据库通过 `drizzle` 对接 `PostgreSQL`。
- 接入约束：必须同时提供 API 文档和 MCP 文档。
- 能力约束：邮箱验证码与手机验证码能力优先使用 Better Auth 官方插件能力，不自研完整 OTP 体系。
- 能力约束：Google 登录优先使用 Better Auth 官方 social provider。
- 能力约束：微信登录优先使用 Better Auth 官方 WeChat provider；当前一期按网页扫码登录能力实现。
- 账号约束：邮箱和手机号是本地主要识别信息，但系统内部应有独立用户主体主键。
- 账号约束：Google 和微信首次登录先创建独立身份，后续通过显式流程合并。
- 合并约束：身份合并支持用户主动触发与后台 / 系统触发两种方式。
- 合并约束：冲突覆盖前必须验证当前用户身份有效，例如具备当前登录方式返回的有效 token。
- 安全约束：作为正式登录系统，账户机制必须完整，且需要确保用户身份信息处理安全。

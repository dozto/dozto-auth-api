# 通用身份认证服务 — 需求说明

## 1. 文档元信息

| 项 | 内容 |
| --- | --- |

## 2. 目标（一句话）

提供**基于 Supabase Auth** 的身份认证能力：客户端**只与本 HTTP 服务**交互，完成登录后获得**访问令牌**（并包含 **refresh token** 等与 Supabase 会话一致的信息），**不向客户端暴露** Supabase 项目 URL、密钥等实现细节。在实现上**优先使用 Supabase Auth 已有数据库与服务**，**尽量少做额外业务开发**。

## 3. 核心范围

| 包含 | 说明 |
| --- | --- |
| 登录与发令 | 完成各类登录流程，并返回与 Supabase Auth 一致的会话 / 令牌结果（含 **access token** 与 **refresh token** 等；对外形状由实现与架构约定，见 `ARCHITECTURE.md`）。 |
| 统一入口 | 所有客户端与集成方**经本服务 HTTP API** 使用认证能力，由本服务封装 Supabase。 |
| 可选校验 | **支持**第三方调用**本服务**对 token 做身份校验；第三方也可**自行**校验 JWT，**不强制**使用本服务。 |

## 4. 登录方式

必须支持以下四类（行为以 Supabase Auth 配置与官方能力为准；**微信**若官方无现成 provider，**允许**在需求范围内做**最小限度**的自定义 OAuth / provider 集成，以满足**网页扫码**流程）：

| # | 方式 |
| --- | --- |
| a | 邮箱 + 一次性验证码（OTP） |
| b | 手机短信 + 一次性验证码（OTP） |
| c | **微信**：仅**网页**场景——页面展示二维码，用户扫码授权后完成登录 |
| d | **Google** 集成登录 |

说明：**不要求**对 Google 账号强制绑定邮箱等业务规则（与「账号」节一致）。

## 5. 用户与账号

- **不**要求用户必须拥有或绑定「已验证」的邮箱或手机号；**不**将「至少一个有效邮箱/手机」作为登录前提，**允许**在未满足该类条件时完成登录（具体用户模型以 Supabase Auth 为准）。
- 身份合并、忘记密码、修改邮箱/手机号、完整「账户中心」类能力**不在本期需求内**（见下节）。

## 6. 非目标（本期不做）

包括但不限于：

- 业务用户中心、资料管理。
- 忘记密码、修改邮箱、修改手机号等账户维护流程（除非后续单独纳入需求）。
- 显式身份合并、冲突处理等产品流程。
- 复杂业务授权模型、本服务作为通用 SSO 提供方。
- **对外 API 文档 / MCP 文档**作为交付物（可后续再加）。
- 在本需求文档中展开日志字段、错误码全集、正式安全/运维白皮书。

## 7. 技术约束（摘要）

- 实现栈与仓库约定见 `agents/ARCHITECTURE.md`（如 `bun`、`hono`、`zod`、`Supabase Auth` 等）。
- 认证状态与持久化**以 Supabase Auth / 托管库为主**；应用层**少造轮子**，非主链路第三方封装放在架构约定目录中。

## 8. 验收要点（服务级）

- 客户端**不直接依赖**暴露 Supabase 连接信息即可完成上述登录方式并取得令牌（含 refresh 相关能力，与 Supabase 行为一致）。
- 四类登录方式在约定环境下可按产品验收（微信为网页扫码路径）。
- 本服务**提供**可被调用的 token 校验能力；第三方**可选用或不用**。
- 不包含第 6 节所列非目标范围内的强制验收。
